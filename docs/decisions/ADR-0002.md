# ADR-0002

# Scope CQRS to Modules with Divergent Read/Write Needs

**Status:** Accepted

**Date:** 2026-07-06

**Decision Makers:**

- Helder Gonçalves

---

# 1. Context

The Software Architecture Document and the Engineering Handbook list CQRS as an architectural pattern used "where applicable", without a concrete rule for when it should be applied.

The first modules scheduled for implementation — Identity (Sprint 02–03), Authorization (Sprint 03), and Master Data (Sprints 04–08: Product, Customer, Supplier, Warehouse, Shared Reference Data) — are, by their own Sprint definitions, straightforward CRUD operations with no meaningfully divergent read and write models.

Without an explicit boundary, there is a real risk of introducing full CQRS (separate read models, Mediator pipelines, dedicated query infrastructure) into modules that do not need it, adding complexity without corresponding value.

---

# 2. Problem Statement

Define a clear, documented rule for when a module should use full CQRS versus a simple Application Service, so that engineers and AI assistants apply the pattern consistently rather than case by case.

---

# 3. Considered Options

## Option 1 — CQRS everywhere from the start

Advantages: consistent pattern across all modules from day one.

Disadvantages: significant boilerplate for simple CRUD modules; contradicts the project's "Simplicity before complexity" principle (Project Charter, Section 6).

## Option 2 — No CQRS at all

Advantages: maximum simplicity.

Disadvantages: modules with genuinely divergent read/write needs (Business Intelligence, complex Sales/Finance reporting) would be forced into an inadequate model, likely requiring a later rewrite.

## Option 3 — Selective CQRS based on read/write divergence

Advantages: matches pattern complexity to actual need; keeps early modules simple; reserves CQRS for where it earns its cost.

Disadvantages: requires an explicit, documented boundary to avoid inconsistent application.

---

# 4. Decision

The ERP Intelligence Platform adopts **Option 3 — Selective CQRS**.

- **Identity and Master Data** (Sprints 02–08) use simple Application Services with Commands and Queries as use-case classes (naming follows Naming Conventions, Section 10), but without separate read models or dedicated query infrastructure.
- **CQRS with dedicated read models** is reserved for modules where read and write concerns genuinely diverge — starting with **Business Intelligence** (EP-008) and any future **Sales/Finance reporting** functionality that requires denormalised or aggregated read models distinct from the transactional write model.
- This boundary shall be revisited, via a new ADR, if a module initially scoped as simple CRUD is later found to need CQRS.

---

# 5. Rationale

This decision aligns with the project's own principles: "Simplicity before complexity" (Project Charter), YAGNI (Cursor Rules, Section 4), and "Quality Over Speed" through avoiding unnecessary abstraction (AI Strategy, Section 4).

It also keeps the first implementation sprints (02–08) focused on delivering working Identity and Master Data modules without the overhead of infrastructure that offers no benefit at that stage.

---

# 6. Consequences

### Positive

- Faster, simpler implementation of the first modules.
- Less boilerplate and cognitive overhead for early sprints.
- CQRS is applied where it provides real value, not everywhere by default.

### Negative

- A module initially treated as simple CRUD may later need to be refactored into CQRS if its read/write needs diverge unexpectedly. This is accepted as a deliberate trade-off; keeping Application layer interfaces clean (Section 4 of this ADR) limits the blast radius of such a refactor.

---

# 7. Implementation Guidelines

- Application layer use cases are still named as Commands and Queries (for example, `CreateCustomerCommand`, `GetCustomerByIdQuery`) regardless of whether full CQRS infrastructure is used, keeping naming consistent across the codebase.
- Simple modules may use a single Application Service class per aggregate; this is not a violation of CQRS naming conventions.
- Business Intelligence and any future reporting module shall introduce dedicated read models and, where justified, a Mediator-based pipeline.

---
