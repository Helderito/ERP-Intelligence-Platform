# ADR-0003

# Defer Multi-Tenancy Beyond the MVP

**Status:** Accepted

**Date:** 2026-07-06

**Decision Makers:**

- Helder Gonçalves

---

# 1. Context

The Software Architecture Document lists multi-company ("Multiempresa") and multi-user support as future scalability goals (Section 16), and the Product Roadmap and README list "Multi-Tenant Architecture" as part of the platform's long-term vision.

Neither the Data Model nor the Domain Model currently include a tenant or company identifier on any aggregate. The MVP (Release 1.0.0) and the Sprints currently planned (00–08) target a single organisation.

A decision is needed on whether to prepare the current Master Data aggregates for multi-tenancy now (for example, adding an unused `CompanyId` field) or to fully defer the concern.

---

# 2. Problem Statement

Decide whether to speculatively prepare the schema and domain model for multi-tenancy ahead of any concrete multi-company requirement, or to defer the decision entirely until multi-tenancy is actually scheduled.

---

# 3. Considered Options

## Option 1 — Reserve a `CompanyId` field now

Advantages: avoids a larger migration later across Master Data and transactional tables.

Disadvantages: designs a multi-tenancy model without a concrete requirement to validate it against — the eventual real design (row-level security, schema-per-tenant, or database-per-tenant) is not yet decided, so a speculative field risks being wrong and needing rework anyway. Also contradicts YAGNI.

## Option 2 — Fully defer multi-tenancy

Advantages: keeps the MVP schema and domain model simple; multi-tenancy is designed properly, as its own Epic, once there is a concrete business need and a chosen tenancy strategy.

Disadvantages: introducing multi-tenancy later will require a non-trivial migration across existing Master Data and transactional tables.

---

# 4. Decision

The ERP Intelligence Platform **defers multi-tenancy beyond the MVP**.

Releases 0.1.0 through 1.0.0 target a single organisation. No tenant or company identifier is added to any aggregate at this stage.

Multi-company support remains a valid long-term vision item (Product Roadmap, README) but is unscheduled until a dedicated Epic formally introduces it, at which point the tenancy strategy (for example, shared schema with row-level filtering, versus schema- or database-per-tenant) shall be decided through a new ADR.

---

# 5. Rationale

This decision follows YAGNI and the "Simplicity before complexity" principle already established in ADR-0002. Designing a multi-tenancy model speculatively, without a concrete requirement to validate it, risks choosing the wrong approach and having to redo it regardless of any groundwork laid now.

---

# 6. Consequences

### Positive

- Simpler schema and domain model for the MVP.
- No wasted design effort on an unvalidated multi-tenancy approach.

### Negative

- Introducing multi-tenancy later will likely require a non-trivial migration across Master Data and transactional tables. This is accepted as a deliberate, documented trade-off rather than an oversight.

---

# 7. Impact

This decision directly affects:

- Data Model
- Domain Model
- Software Architecture Document (Section 16, Scalability)
- Product Roadmap (Multi-Tenant Architecture remains a long-term item, unscheduled)

---

# 8. Alternatives for Future Review

When multi-company support is scheduled, a new ADR shall define the chosen tenancy strategy before any implementation begins.
